Isolation Levels

in MySQL / MariaDB

A look at transaction isolation — what anomalies exist, how MySQL prevents them at each level, and why the right choice depends on your app.

ACIDtransactionsconcurrencyMySQL 8.0MariaDB 10.x

SAY
Today we're presenting on isolation levels in MySQL and MariaDB. Isolation is the "I" in ACID — it controls how transactions behave when running at the same time. We'll cover the main anomalies that can go wrong, how each isolation level handles them, and why MySQL doesn't just always use the safest option.

What is Isolation?

/** Isolation = "I" in ACID * Each tx runs as if alone. */ interface ACID { Atomicity: "all or nothing"; Consistency: "valid state"; Isolation: "no interference"; // ← Durability: "persisted"; }
Without isolation, concurrent transactions can read and write the same data — leading to corrupted results.
⚠ Race condition: bank withdrawal
T1 reads balance = $5,000
T2 reads balance = $5,000
T1 withdraws $5,000 ✓
T2 withdraws $5,000 ✓
Final balance = -$5,000 💀
→ Isolation prevents this.
SAY
Isolation means each transaction runs as if it's the only one — even if hundreds are running simultaneously. Without it, you get race conditions like this bank example — both transactions see the same balance, both withdraw, and the account goes negative. That's a real production bug that isolation is specifically designed to prevent.

SAY
There are three main anomalies that happen without proper isolation. A dirty read is reading uncommitted data — if that transaction rolls back, you acted on data that never really existed. A non-repeatable read is when the same query returns different values within one transaction because another tx changed the row. A phantom read is when new rows appear in your result set mid-transaction. Each isolation level is designed to prevent these one by one.

The 4 Isolation Levels

★ MySQL / MariaDB default  ·  InnoDB + MVCC
SAY
SQL defines four isolation levels. Read Uncommitted is the loosest — all three anomalies are possible but performance is highest. Each level up prevents one more anomaly. Repeatable Read is MySQL's default — highlighted here — it handles dirty and non-repeatable reads. Serializable prevents everything but runs transactions one at a time, which kills performance for high-traffic apps.

MySQL default
Defaults to REPEATABLE READ — the sweet spot between safety and speed for most web apps using InnoDB.
MVCC
Multi-Version Concurrency Control — reads don't block writes, so MySQL is safer without sacrificing as much performance.
SAY
We need multiple levels because there's a tradeoff between safety and performance — higher isolation prevents more bugs but also means fewer transactions can run at the same time. MySQL's default Repeatable Read works because it uses MVCC, which lets reads and writes happen in parallel without blocking each other — so you get decent safety without tanking your throughput.

SAY
To wrap up — isolation protects concurrent transactions from interfering with each other. MySQL gives us four levels depending on how much safety vs speed we need, and you set it with SET SESSION TRANSACTION ISOLATION LEVEL. For most apps the default Repeatable Read is the right call — but for something critical like banking, you'd want Serializable. Any questions?
