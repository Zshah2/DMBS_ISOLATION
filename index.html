Isolation Levels

in MySQL / MariaDB

When multiple users query and update a database at the same time, things can go wrong. Isolation levels define the rules that prevent transactions from interfering with each other — and choosing the right one is a critical decision in any real-world database system.

SAY
Good [morning/afternoon], today we're presenting our DBMS project on isolation levels in MySQL and MariaDB. Isolation is one of the four ACID properties — it's what protects your data when multiple users hit the database at the same time. We'll start by explaining what isolation actually means, walk through the three problems that happen without it, cover the four levels MySQL gives us, and explain why choosing the right one is a real engineering decision.

What is Isolation?

/** * The "I" in ACID. * Guarantees that concurrent * transactions execute as if * they were serialized. */ interface ACID { Atomicity: "all or nothing"; Consistency: "always valid state"; Isolation: "no interference"; // ← Durability: "committed = permanent"; }

In simple terms: Even when 1,000 transactions are running at the same time, each one should behave as if it is the only transaction in the system. It should not be able to see half-finished work from another transaction.

WHY IT MATTERS

Without isolation
Two transactions reading and writing the same row at the same time can corrupt data, produce wrong results, or make one transaction invisible to the other — even when both succeed.
With isolation
Each transaction works with a consistent snapshot of the data. Changes from other transactions are either fully visible or fully hidden — never partial.
SAY
Isolation is the "I" in ACID. The idea is simple — every transaction should behave as if it's the only one running, even if thousands are happening simultaneously. Without isolation, two transactions reading and writing the same data at the same time will get unpredictable results. With isolation, each transaction works from a consistent, stable view of the database — and the level of isolation controls exactly how strict that guarantee is.

The Concurrency Problem

real world
⚠ Scenario: bank withdrawal
Account balance = $5,000
T1 reads balance → $5,000
T2 reads balance → $5,000
T1 withdraws $5,000 → balance = $0
T2 withdraws $5,000 → balance = -$5,000 💀

Both transactions read the balance before either commits. Both see $5,000. Both succeed. The account is now overdrawn. This is not a bug in the application logic — it's a database concurrency problem.

WHAT WENT WRONG

No locking or isolation
T1 and T2 both read the same value before either one writes. Neither knows the other exists. This is called a lost update — one transaction's change overwrites the other's.
Real-world impact
Banking, e-commerce, booking systems, inventory management — any system with concurrent writes to shared rows is vulnerable without proper isolation in place.
FIX →
Isolation forces T2 to wait until T1 commits, then read the updated balance of $0 — preventing the overdraft entirely.
SAY
Here's a concrete example of what can go wrong without isolation. Two users withdraw $5,000 from the same account at the same time. Both read the balance as $5,000 before either one has committed. Both think the withdrawal is valid. Both succeed. The account ends up at -$5,000. This isn't a bug in the app code — it's a database concurrency problem. Proper isolation prevents this by ensuring T2 can't read the old balance until T1 finishes.

Anomaly 1: Dirty Read

anomaly

Definition: A dirty read occurs when Transaction A reads data that Transaction B has written but not yet committed. If B rolls back, A has acted on data that never officially existed in the database.

t=1START TX
t=2WRITE $800 (uncommitted)
t=3READ → $800 ← dirty!
t=4Acts on $800ROLLBACK
t=5ERROR: ghost databalance = $1,000
Step-by-step
1/5
Transaction 1 (T1)
START TRANSACTION
READ balance → $800 (dirty!)
Process withdrawal of $800
COMMIT — based on ghost data
Transaction 2 (T2)
WRITE balance = $800 (uncommitted)
ROLLBACK → balance = $1,000
T1 starts a transaction. T2 has not yet made any changes.
FIXED BY →
READ COMMITTED and above. At READ COMMITTED, T1 can only read data that has been committed — so it never sees T2's $800 until T2 commits.
SAY
The first anomaly is the dirty read. T2 writes a new balance of $800 but hasn't committed yet. T1 reads that value — $800 — and starts processing based on it. Then T2 rolls back, reverting the balance to $1,000. T1 just acted on data that never actually existed in the database. This is called dirty data. It's fixed by READ COMMITTED — which only allows reading committed data.

Anomaly 2: Non-Repeatable Read

anomaly

Definition: A non-repeatable read occurs when Transaction A reads the same row twice within one transaction and gets different values — because Transaction B modified and committed that row between A's two reads.

TIMELINE
timeT1T2
t=1READ price → $50
t=2UPDATE price = $75
t=3COMMIT
t=4READ price → $75 ≠ $50!
Step-by-step
1/5
Transaction 1 (T1)
START TRANSACTION
READ price → $50
READ price → $75 ≠ $50!
Inconsistency within same tx!
Transaction 2 (T2)
UPDATE price = $75
COMMIT ✓
T1 starts and is about to read a product price.
FIXED BY →
REPEATABLE READ and above. MySQL's default. T1 is guaranteed to see the same value for any row it has already read, for the entire duration of the transaction.
SAY
The second anomaly is the non-repeatable read. T1 reads a product price — $50 — and does some calculations. Meanwhile T2 updates that price to $75 and commits. When T1 reads the same row again, it gets $75 — a different value than before. Within one single transaction, the same query returned two different results. That breaks the assumption that data stays stable during your transaction. It's fixed by REPEATABLE READ — MySQL's default.

Anomaly 3: Phantom Read

anomaly

Definition: A phantom read occurs when Transaction A runs the same range query twice within one transaction and gets a different set of rows — because Transaction B inserted or deleted rows that match the query's condition in between.

TIMELINE
timeT1T2
t=1COUNT orders → 5 rows
t=2INSERT new order (id=6)
t=3COMMIT
t=4COUNT orders → 6 rows!
Step-by-step
1/5
Transaction 1 (T1)
START TRANSACTION
COUNT orders → 5 rows
COUNT orders → 6 rows!
Phantom row appeared!
Transaction 2 (T2)
INSERT new order (id=6)
COMMIT ✓
T1 starts and counts all orders in the database.
FIXED BY →
SERIALIZABLE only. This is the hardest anomaly to prevent — it requires locking entire ranges of rows, not just individual ones, which is why it needs the strictest level.
SAY
The third anomaly is the phantom read. T1 counts 5 orders. T2 inserts a new order and commits. T1 counts again and now sees 6. A row that didn't exist before has appeared like a phantom. Notice this is different from non-repeatable reads — here it's not the data inside a row that changed, but the number of rows in the result set. This is the hardest to prevent and requires SERIALIZABLE — the strictest isolation level.

The 4 Isolation Levels

isolation-matrix.sql
Level Dirty Read Non-Repeatable Phantom Read Use Case Performance
READ UNCOMMITTED ✗ possible ✗ possible ✗ possible Rarely used — analytics only highest
READ COMMITTED ✓ prevented ✗ possible ✗ possible PostgreSQL default, reporting high
REPEATABLE READ ★ ✓ prevented ✓ prevented ✗ possible* MySQL default — most web apps medium
SERIALIZABLE ✓ prevented ✓ prevented ✓ prevented Financial, audit, compliance lowest
★ MySQL / MariaDB default  ·  * InnoDB gap locks partially prevent phantom reads under REPEATABLE READ
Reading down the table
Each level adds one more protection. As you go down, more anomalies are prevented but at a cost — more locking, less concurrency, lower throughput.
Why not always use SERIALIZABLE?
At SERIALIZABLE, transactions queue up one by one. For high-traffic apps with thousands of concurrent users, this kills performance. The tradeoff is real.
SAY
Here's the full picture — four isolation levels, each one preventing one more anomaly than the last. Read Uncommitted is the weakest — all anomalies are possible. Read Committed stops dirty reads. Repeatable Read — MySQL's default — also stops non-repeatable reads. Serializable stops everything. The extra column here is use case — notice MySQL defaults to Repeatable Read because it's the right balance for most web apps. The footnote is interesting too — InnoDB's gap locks mean Repeatable Read actually prevents some phantom reads in practice, even without going full Serializable.

The engineering decision
Choosing an isolation level is always a tradeoff. A social media feed can tolerate slightly stale reads. A bank transfer cannot. That's why MySQL gives you control — there is no one-size-fits-all answer.
SAY
So why don't we just always use Serializable? Because of the bars you see here — safety and speed pull in opposite directions. At Serializable, transactions queue up one at a time, which means fewer can run simultaneously and throughput drops. For a banking app that's fine — correctness is everything. But for a high-traffic social feed or e-commerce site serving millions of users, that bottleneck kills performance. The right level depends on what your application actually needs.

MySQL / MariaDB Specifics

internals
1-- Check current isolation level 2SELECT @@transaction_isolation; 3 4-- Set for current session only 5SET SESSION TRANSACTION 6 ISOLATION LEVEL REPEATABLE READ; 7 8-- Set globally (all new sessions) 9SET GLOBAL TRANSACTION 10 ISOLATION LEVEL READ COMMITTED; 11 12-- Use inside a transaction 13START TRANSACTION; 14SELECT balance FROM accounts WHERE id=1; 15-- ... your logic ... 16COMMIT;
Note: older MySQL / MariaDB → use @@tx_isolation
Default: REPEATABLE READ
MySQL and MariaDB both default to Repeatable Read — the sweet spot for most web applications using InnoDB.
InnoDB engine required
Isolation levels only apply to InnoDB. The older MyISAM engine has no transaction support — it ignores isolation entirely.
MVCC — reads don't block writes
MySQL uses Multi-Version Concurrency Control — readers see a snapshot of the data, so reads never block writes. This is how MySQL achieves safety without sacrificing too much speed.
Gap locks under REPEATABLE READ
InnoDB uses gap locks to lock ranges between rows — this partially prevents phantom reads even at Repeatable Read, which is stronger than the SQL standard requires.
SAY
Here's how you actually use this in MySQL. You check your current level with SELECT @@transaction_isolation, and set it with SET SESSION TRANSACTION ISOLATION LEVEL before starting your transaction. A few MySQL-specific things worth knowing — isolation only works on the InnoDB engine, not MyISAM. MySQL uses MVCC which means readers never block writers — that's why Repeatable Read is fast enough for most apps. And InnoDB's gap locks actually give you stronger phantom read protection than the SQL standard requires at that level.
